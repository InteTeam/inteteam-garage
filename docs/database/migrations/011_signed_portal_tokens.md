## Purpose

Backs the per-job customer portal URL. Each token is a long, opaque string the customer visits at `portal/{token}`. The `ValidatePortalToken` middleware looks the token up, checks `expires_at` and `revoked_at`, and binds the matching `RepairJob` to the route. Tokens are short-lived; mechanic can regenerate via `PortalLinkController@regenerate` (which revokes the previous one).

## Model Configuration

- Class: `App\Models\SignedPortalToken` (final)
- Traits: `HasUlids`, `HasGarageScope`, `HasFactory`
- Policy: **none** — token validation is the access mechanism itself; no separate authorization layer

- `casts()`: `expires_at => datetime`, `revoked_at => datetime`

## Constraints & Invariants

- `token` is generated by `SignedPortalTokenService::generate()` (Str::random with sufficient entropy). Never accept a client-supplied token value.
- Regenerating a token sets `revoked_at = now()` on the previous row and inserts a new one — old rows stay for audit, never deleted.
- Middleware rejects requests where `expires_at < now()` OR `revoked_at IS NOT NULL`.
- Token includes `garage_id` so a leaked token from garage A cannot be replayed against a job in garage B (defence-in-depth — single-tenant logic should already prevent).

## Deviations from playbook

- **No `idx_garage_created`.** Token lookups are by `token`, not by garage list. Acceptable; tenant-scoped UI for token history is not a current use case.
