# 009 — repair_job_mechanic (pivot)

**Migration file:** `database/migrations/20260520200450_create_repair_job_mechanic_table.php`
**Added:** 2026-05-20
**Phase:** 1 — Multi-Tenancy & Core Models

## Purpose

Many-to-many pivot for assigning mechanics to repair jobs. One job can have multiple mechanics working on it; one mechanic can be assigned to many jobs. Assignment is tracked here, not on `repair_jobs` or `mechanics` directly. Manual migration — **not generated by `integen`** per `docs/tasks.md` Phase 1.

## Schema

| Column | Type | Null | Default | Notes |
|---|---|---|---|---|
| `repair_job_id` | ULID | no | — | FK → `repair_jobs.id` cascadeOnDelete; part of composite PK |
| `mechanic_id` | ULID | no | — | FK → `mechanics.id` cascadeOnDelete; part of composite PK |
| `created_at`, `updated_at` | timestamp | yes | — | Pivot timestamps |

## Indexes

| Name | Columns | Purpose |
|---|---|---|
| Composite PK | `(repair_job_id, mechanic_id)` | Enforces one assignment per (job, mechanic) and serves both directions |

No `id` column — the composite primary key is sufficient.

## Foreign Keys

- `repair_job_id` → `repair_jobs.id` (ULID, `cascadeOnDelete`)
- `mechanic_id` → `mechanics.id` (ULID, `cascadeOnDelete`)

Deleting either side removes the assignment automatically.

## Relationships

```
RepairJob ──belongsToMany──► Mechanic   (via repair_job_mechanic)
Mechanic ──belongsToMany──► RepairJob   (via repair_job_mechanic)
```

- `RepairJob::mechanics(): BelongsToMany` → `belongsToMany(Mechanic::class, 'repair_job_mechanic')`
- `Mechanic::repairJobs(): BelongsToMany` → `belongsToMany(RepairJob::class, 'repair_job_mechanic')`

## Model Configuration

No standalone Eloquent model — this is a pure pivot table accessed via the `BelongsToMany` relations on `RepairJob` and `Mechanic`.

## Constraints & Invariants

- Composite PK guarantees a (job, mechanic) pair appears at most once.
- No `garage_id` on the pivot — tenant isolation is inherited transitively: both `RepairJob` and `Mechanic` carry `HasGarageScope`, so cross-garage joins are impossible through the application path. A direct SQL query joining the pivot would bypass this — keep it to Eloquent.
- No soft-deletes on the pivot — un-assigning is a hard delete from the pivot row. The audit trail of who-was-assigned-when, if needed, must be tracked separately (e.g. an `ApprovalEvent` row).
- Pivot timestamps are present (`created_at`, `updated_at`) — useful for "assigned on" UI without needing a separate audit hop.

## Deviations from playbook

- **No `garage_id` column on the pivot.** Playbook recommends tenant scoping on every multi-tenant table. The trade-off here is the composite PK + dual cascadeOnDelete from `repair_jobs` (which IS tenant-scoped) gives effective scoping. Acceptable — but means raw SQL bypasses the scope. Document this as the contract.
